# Bases vs Markdown

**Правило:** Outlook на инструмент учёта зависит от частоты изменений и потребности в фильтрах.

- **Obsidian Bases** оправдан, когда:
  - Изменения **редкие** (раз в недели/месяцы)
  - Каждая запись несёт **много полей** (категория, статус, бренд, размер, локация, дата)
  - Нужны **фильтры/views** (купить, сломано, по локации)
  - Записи могут **связываться** с другими (рецепт → оборудование)
  - Пример: оборудование, мебель, инструменты, контакты-CRM
- **Markdown-таблица** оправдан, когда:
  - Изменения **частые** (ежедневно/еженедельно)
  - Каждая запись — простая (название, кол-во, заметка)
  - Достаточно одной плоской таблицы / одного списка
  - Источник истины — текстовый, читается агентом
  - Пример: продукты, текущий список покупок, daily metrics

**Why:** В сессии 2026-05-01 (WP-2 Питание) разделили: оборудование → Bases (`Vault1/Питание/Кухня DB.base`, 19 заметок), продукты → markdown-таблица (`DS-strategy/inbox/WP-2-inventory.md`). Причина — оборудование меняется раз в годы и имеет 7-10 полей с фильтрами, продукты меняются еженедельно с 3-4 полями. Смешение в один формат = либо Bases-перегруз для простых списков, либо md-перегруз когда нужны фильтры.

**How to apply:**
- При запросе «давай создадим учёт X» — спросить: сколько меняется в неделю? сколько полей у одной записи? нужны ли фильтры?
- ≤3 поля + еженедельные изменения → markdown.
- ≥5 полей + редкие изменения + нужны фильтры → Bases.
- Гибрид допустим: оборудование в Bases, продукты в md, ссылки между ними через путь.
- Не путать с «Pack-знание ≠ Реализационное решение» (HD #29) — выбор формата = реализационное решение, не доменное знание.
