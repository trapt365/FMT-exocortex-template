# DayPlan inline-update при закрытии пункта

**Правило:** при закрытии любого пункта DayPlan (статус → done / drop / carry-over) — обновить строку в `DS-strategy/current/DayPlan YYYY-MM-DD.md` **сразу в момент закрытия**, не откладывать до Quick Close.

**Why:** 9 мая 2026 пользователь обнаружил, что в соседнем чате `открываем рабочую сессию` AskUserQuestion предлагал РП, которые уже отработаны в этой сессии (Diesel-gen, memo hotel, аудио 9 мая). Причина: 4 пункта DayPlan закрыты в течение сессии, но файл не обновлён. Соседний чат читал stale DayPlan из git и предлагал «незакрытые» пункты, среди которых уже была проделана большая работа.

Quick Close (правило S-16) обновляет DayPlan только в момент формального закрытия сессии. Между чатами / параллельными сессиями / при срабатывании `notify.sh` — DayPlan остаётся в утренней редакции, что создаёт рассинхрон.

**How to apply:**

1. **Триггер inline-update:** любое из:
   - Пользователь подтверждает результат («сделано», «закрыто», «отправлено», «готово»)
   - Закоммичен ключевой артефакт пункта (session-prep, context-file, code change, отчёт)
   - Принят drop / carry-over («ждать», «на завтра», «не сегодня»)

2. **Что обновить в DayPlan:**
   - Эмодзи статуса (✅ / 🟡 carry / 🔴 missed / 🟢 drop)
   - Колонка «Статус» → `done` / `carry` / `dropped`
   - Колонка «Результат» → 1-2 строки фактического итога с ссылкой на коммит (если есть)
   - Если ad-hoc (вне плана) — добавить новую строку с ✅ и пометкой `ad-hoc`

3. **Закоммитить.** Один маленький коммит лучше отложенного большого. Сообщение: `DayPlan YYYY-MM-DD: <что закрылось>` или батч `sync статусов`. Push сразу — иначе всё равно stale в other clones / соседнем чате.

4. **Quick Close в конце сессии — финальная сверка**, не первый sync. Чеклист Haiku R23 проверит формальное соответствие, но статусы уже свежие.

**Исключения:** если пункт `pending` остаётся в работе и закроется через 5-10 минут — можно один раз батчем. Но не «накопить за полдня».

**Связанные правила:** S-16 (Quick Close обновляет DayPlan), S-15 (бюджет 👤/🤖 при закрытии). Это правило строже — не «при закрытии сессии», а «при закрытии пункта».

**Промоция:** кандидат на staging S-NN. Если 2 недели применяется без сбоев — занести в STAGING.md как `validated`, потом в FMT.
