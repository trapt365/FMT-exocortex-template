Формулировка задач в Singularity: обезьянопонятное действие (глагол) + рабочий продукт в скобках.

**Why:** методология Дорофеева — задача должна быть понятна «обезьяне» (конкретное действие), а рабочий продукт фиксирует ожидаемый результат. Без глагола и наблюдаемого РП нет критерия выполнения.

**How to apply:**
- Название: `Глагол [объект] (РП: наблюдаемый артефакт или состояние)`
- Примеры:
  - ✅ `Написать onboarding stories 0-2 (user stories в трекере)`
  - ✅ `Загрузить транзакции в Google Sheet (РП: актуализированная таблица)`
  - ✅ `Принять решения по подпискам (РП: список к отмене + план действий)`
  - ❌ `Трекинг MVP: story 0-2 onboarding`
  - ❌ `Финансовая осознанность — учёт, подписки, путь к 25 млн`
- Задачи берутся из полей **Работа** и **Метод** Ритуала согласования (не из названия РП)
- Начинать с простейшего первого шага — декомпозировать если нужно
- Если задача комплексная: одна родительская + подзадачи чеклистом в `note` (API Singularity не поддерживает subtasks нативно)
- Последовательное подтверждение перед созданием — предложить декомпозицию, дождаться подтверждения
- **Проверка перед созданием (ОБЯЗАТЕЛЬНО):** прогонять каждую формулировку через 3 критерия: (1) глагол наблюдаемый и активный (не «провести», «прокачать»); (2) РП в скобках = артефакт результата, НЕ название РП и НЕ описание входа; (3) прилагательные/наречия с операциональным определением. Источник: [[feedback-dorofeev-goal-cards]]
- **По умолчанию — во Входящие, НЕ в Сегодня (2026-06-01):** при `create-task` НЕ задавать `startDate` (без даты → задача падает во «Входящие» Singularity). `startDate` ставить ТОЛЬКО когда задача явно планируется на конкретный день (пользователь сказал «на сегодня/завтра» или это слот расписания). **Why:** заказчик сам раскладывает задачи из Входящих по дням — агент не должен принудительно ставить всё на «Сегодня» (создаёт ложное расписание + давление). Переопределяет дефолт Day Open §8b / Singularity Gate (`startDate=сегодня`). **How:** массовый push РП недели / WP Gate → без `startDate`; явный дневной слот → со `startDate`.
- **Дедуп перед create-task (ОБЯЗАТЕЛЬНО, 2026-06-01):** ДО создания задачи — `list-tasks` по **проекту** назначения, не только `list-tasks-today`. Сверить по ключевым словам с уже существующими. Протокол Day Open §8b это требует, но легко пропустить → создаются дубли поверх существующего backlog. Кейс: 1 июня создал «Активировать Фазу A» поверх уже существовавших WP-32-задач в проекте «Дети» (проверил только today-список). Если задача РП уже есть в проекте — **обновить** её (startDate/useTime/timeLength) на сегодняшний слот, а не плодить новую. **Why:** OwnerIntegrity — один РП = одна задача в проекте; дубли ломают учёт и захламляют backlog. **Календарь ≠ Singularity (смысловой дубль, 2026-06-01):** НЕ создавать задачу-зеркало события календаря. Событие («забрать детей», «созвон», «оплатить X», «открывается регистрация») уже трекается в календаре — отдельная задача с тем же смыслом = смысловой дубль, даже если формулировка чуть иная. Прежде чем счесть задачу «доп. ценностью» (Kaspi-ссылка, окно времени) — **прочитать описание самого события**: обычно ценность уже там (у футбольной оплаты Kaspi-ссылка была в description события). Singularity-задачу создавать только если это отдельная РАБОТА (РП / подготовка / follow-up), не покрытая событием. В DayPlan события показывать как 🔒-якоря расписания (для когнитивной раскладки), но НЕ пушить их в Singularity. **Также — рекуррентные дубли:** просроченные неотмеченные инстансы повторяющихся задач (`...-YYYYMMDD`) накапливаются в today-списке — удалять старый инстанс, оставлять текущий.
