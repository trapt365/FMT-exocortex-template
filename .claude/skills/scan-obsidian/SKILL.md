# Scan Obsidian Daily Notes

Сканирует daily notes из Obsidian vault и извлекает полезную информацию для IWE.

## Конфигурация

- **Vault:** `/mnt/c/Users/Timur/Documents/Vault1/`
- **Daily notes:** `Calendar/YYYY-MM-DD.md`
- **Период:** $ARGUMENTS дней (по умолчанию 7)

## Алгоритм

### Шаг 1. Собрать daily notes

1. Определи период: сегодня минус N дней
2. Прочитай все `Calendar/YYYY-MM-DD.md` за период
3. **Фильтрация шума:** Игнорируй:
   - Блоки `` ```dataview `` и `` ```dataviewjs `` (до закрывающих `` ``` ``)
   - Блоки `> [!info]-` (callout с dataview)
   - Шаблонные секции без заполнения (пустые `- [ ]`, незаполненные промпты)
   - Ссылки на вчера `[[YYYY-MM-DD|вчера]]`
4. Извлеки содержимое из секций:
   - **«Возвращаемся во вчера»** — события и инсайты (если заполнены)
   - **«Что надо точно сделать сегодня»** — задачи
   - **«Log (дела)»** — основной контент: что делал, с кем говорил, что понял
   - **«Scratchpad»** — свободные мысли

### Шаг 2. Классифицировать находки

Для каждой значимой записи определи тип:

| Тип | Признак | Пример |
|-----|---------|--------|
| **НЭП** | Жалоба, недовольство, «не нравится», «бесит», «надо бы» | «недоволен практикой дневного планирования» |
| **Апдейт по проекту** | Упоминание проекта (stomresearch, kense, Геонлайн, meal prep, СМ) | «Прочитал две главы про предметы интереса» |
| **Инсайт / знание** | Различение, паттерн, осознание, фидбек от эксперта | «Разница: предмет интереса vs характеристика vs интерес» |
| **Задача** | Конкретное действие, которое нужно сделать | «Позвонить врачу» |
| **Шум** | Рутина без информационной ценности | «Проснулся, попил кофе» |

### Шаг 3. Маршрутизация

Для каждой находки (кроме шума) определи куда:

| Тип | Куда | Действие |
|-----|------|----------|
| НЭП (новая) | `DS-strategy/docs/Dissatisfactions.md` | Добавить строку в «Активные» |
| НЭП (существующая обновлена) | `Dissatisfactions.md` | Обновить комментарий |
| Апдейт по проекту | `DS-strategy/inbox/WP-{N}-*.md` | Обновить «Текущее состояние» |
| Инсайт / знание | `DS-strategy/inbox/fleeting-notes.md` | Добавить как capture-кандидат |
| Задача → к существующему РП | `DS-strategy/inbox/fleeting-notes.md` | Привязать к РП |
| Задача → новая | `DS-strategy/inbox/fleeting-notes.md` | Предложить создать РП |

### Шаг 4. Контекстная сверка

Перед предложением:
1. Прочитай текущий `Dissatisfactions.md` — не дублировать существующие НЭП
2. Прочитай текущий `WeekPlan` — привязать апдейты к активным РП
3. Прочитай `WP-REGISTRY.md` — найти номер РП по проекту

### Шаг 5. Отчёт

Покажи пользователю:

```markdown
## Obsidian Scan Report

**Период:** YYYY-MM-DD — YYYY-MM-DD
**Просмотрено:** N daily notes

### Находки

| # | Дата | Тип | Текст | Куда | РП |
|---|------|-----|-------|------|-----|
| 1 | ... | НЭП / апдейт / инсайт / задача | краткое содержание | файл | #N |

### Новые НЭП (если есть)
- [описание] → Dissatisfactions.md

### Апдейты по проектам
- WP-N: [что изменилось]

### Captures (кандидаты для Pack)
- [знание] → PACK-{domain} ({тип})

### Не обработано (шум)
- N записей пропущено
```

### Шаг 6. Применение

1. Дождись одобрения пользователя
2. Для каждой одобренной находки — записать в целевой файл
3. Не изменять файлы Obsidian vault (read-only)

## Что НЕ делать

- Не модифицировать файлы в Obsidian vault
- Не дублировать уже существующие НЭП
- Не создавать РП без одобрения (только предлагать)
- Не извлекать dataview-скрипты как контент
